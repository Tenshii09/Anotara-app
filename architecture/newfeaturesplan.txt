Here is the exact API System Connection and Modification Plan to bridge the database upgrades to the user interface and the Machine Learning engine.
Phase 1: The Context-Aware Generation Engine (API Core)
The initial trip generation must ingest the new demographic and logistical variables to feed the ML clustering algorithm.
Endpoint: POST /api/v1/itineraries/generate
System Logic:
Data Ingestion: The API receives the payload including pacing_style, companion_type, transport_mode, and the geocoded accommodation_lat/lng.
Hard Filtering (The Pre-processing Stage): Before the ML model even runs, the backend executes strict SQL exclusions. If companion_type is 'Family_Kids' or 'Seniors', it instantly filters out all rows in the places table where physical_intensity = 'High'.
Route Anchoring: The accommodation_lat/lng is set as "Node 0" in the distance calculation matrix. All Day 1 routing begins and ends relative to this coordinate.
Pacing Governor: The ML clustering algorithm adjusts its density parameter based on pacing_style. If 'Relaxed', it caps sequence_order limits to 2-3 stops per day_number and increases the estimated_duration buffers.
Phase 2: The Granular Control API (Interactive UI Support)
To support the drag-and-drop and micro-editing features without crashing the system or forcing full regenerations, we need modular, lightweight endpoints.
Drag-and-Drop Reordering:
Endpoint: PATCH /api/v1/itineraries/{id}/items/reorder
System Logic: When a user drags an item in the frontend, React sends an array of item_ids and their new sequence_order integers. The backend performs a batch UPDATE in the itinerary_items table, instantly recalculating the polyline map coordinates to send back to the UI.
The Micro-Swap Engine:
Endpoint: POST /api/v1/itineraries/{id}/items/{item_id}/swap
System Logic: The system identifies the geographic cluster of the rejected place. The ML model queries the places table for the next-highest scored location within a 3-kilometer radius that shares the same budget and environment_type. It increments swap_history by +1 for the rejected item and replaces the record.
The State Locker:
Endpoint: PATCH /api/v1/itineraries/items/{item_id}/lock
System Logic: Toggles the is_locked boolean. When a user subsequently hits the "Regenerate Day" endpoint, the database script runs DELETE FROM itinerary_items WHERE day_number = X AND is_locked = FALSE, safely preserving their anchored choices.
Phase 3: The Automated "Live Monitor" & Webhooks
This is the system integration masterstroke that ties external APIs, database state, and mobile notifications together.
The Cron Job (Background Task):
System Logic: A scheduled background worker script runs every 6 hours. It queries the itineraries table for records where status = 'Active' and the start date is within 72 hours.
External API Integration:
The worker pulls the lat/lng of the itinerary and hits an external Weather API (e.g., OpenWeatherMap).
The Pivot Trigger:
If the Weather API returns a high probability of precipitation, the system flags the itinerary. It pre-calculates an alternative route by exclusively querying the places table where environment_type = 'Indoor'.
It triggers a push notification payload to the React PWA via Firebase Cloud Messaging, prompting the user to review the "Smart Suggestion."
Phase 4: The Machine Learning Retraining Loop
The feedback module must be decoupled from the core UX to prevent app slowdowns.
Endpoint: POST /api/v1/feedback
System Logic: The UI sends the rating_type ('Best Pick', 'Not Ideal') seamlessly in the background. The API writes this to the trip_feedback table.
The Evolution Script: During off-peak hours (e.g., 3:00 AM), an automated Python script aggregates the trip_feedback data. If a specific place_id accumulates a high volume of 'Not Ideal' ratings or has a massive spike in swap_history, the script automatically downgrades its baseline recommendation weight in the primary ML model.

2. Upgrading the itinerary_items Table (The Granular Engine)
This table links the saved places to a specific itinerary. To support Drag-and-Drop Reordering, Granular Swapping, and the Interactive Day-by-Day View, we must add strict sequencing and locking mechanisms.
Required New Columns:
day_number (INT): Groups the stops (e.g., Day 1, Day 2) for the frontend tabbed view.
sequence_order (INT): Determines the chronological order of stops within a specific day. If a user drags and drops a place, this number updates.
estimated_duration (INT): How many minutes the user is expected to spend here (creates the "Breathing Room" pacing).
is_locked (BOOLEAN): A crucial UX feature. If a user regenerates a day but wants to keep a specific restaurant, they "lock" it. The ML system will skip this row when deleting/replacing items for that day.
swap_history (INT): Tracks how many times a user clicked "Swap This Place". Useful for analytics to see if the model's first choice was poor.
3. Upgrading the places (or destinations) Table (The ML Fuel)
To allow the system to pivot during a Live Trip Monitor (Weather Alert) or when filtering for Companion Types, the actual places in your database need metadata.
Required New Columns:
environment_type (ENUM): 'Indoor', 'Outdoor', 'Mixed'. If the background task detects rain via the weather API, the ML model will exclusively query places marked 'Indoor'.
physical_intensity (ENUM): 'Low' (Museums, Cafes), 'Medium' (Walking tours), 'High' (Hiking, Surfing). Filtered against the user's companion_type.
